# 线程池参数设计

### 业务背景
- 日活用户数20万,订单近100万。
- 业务高峰时段: 日志显示10分钟内有2500用户下过单。
### 服务器背景
- CPU核数：12

### query层
- 线程数为2倍cpu核数24
- 阻塞队列长度为60
- query层如果根据公式"cpuNum * cpuUse * (1 + W/C)"计算线程数的值是偏大，考虑上下文切换开销，先从cpuNum*2开始测。后续可考虑增加。
- 拒绝策略:CallerRunsPolicy。天然背压，降低上游拉取MQ消息的速度。
- 外部调用超时时间为500ms.提交下游有指数退避重试机制，最大时间为700ms(100ms+200ms+400ms),极端情况下query层每秒处理任务数为24*(1s/1.2s) = 20个/s。 队列任务最大延迟为60/20=3s
- 假定正常工作时query层任务耗时为200ms。则吞吐量为24*(1s/0.2s) = 120个/s,可以支撑峰值流量。

### merge层
- 长度12的单线程池数组
- 每个单线程池的阻塞队列长度为20，
- 拒绝策略AbortPolicy，触发时上游query层降低速度。

### write层
- 队列长度12,因为上游每分钟提交12个List集合。write层希望保持严格模式，不进行任务堆积。
- 单线程批量写入,每次写入200条。假定单次写入耗时50ms，每分钟可写入理论上限是```200条 * (1s/50ms) = 4000条/s```。远超峰值10分钟2500个用户。
- 拒绝策略AbortPolicy。上游继续聚合，由于聚合的是订单数，只有用户增加才会增大记录数，并不会导致数据堆积，反而提高了聚合度，只是略微增大数据延迟。

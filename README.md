# 1.背景与挑战
小程序日均快递订单百万，需要统计用户每日订单数，给运营团队做数据分析支持。

# 2. 原始方案
### —— 基于共享 ConcurrentHashMap 的多线程聚合方案。

##### 思路: 采用多线程在内存中对数据做聚合再入库的思路。
##### 具体实现：
| 角色       | 职责            | 实现         |
| -------- | ------------- | ---------- |
| 生产者（多线程） | 将订单数据聚合写入共享集合 | 并发消费 MQ 消息 |
| 消费者（单线程） | 从集合取出聚合结果落库   | 定时任务触发     |


##### 共享集合设计：
- 类型：ConcurrentHashMap
- key：数据库唯一索引条件，保证聚合唯一性
- value：使用AtomicInteger实现多线程累加

##### 并发控制：
- 生产者与消费者共享集合，通过**读写锁**隔离：生产者加读锁，消费者加写锁
- 用**写时复制思想**降低写锁持有时间：消费者加锁后将公共引用切换到新空 Map 即可释放锁，旧 Map 由方法内局部引用持有完成落库
##### 技术点：多线程、ConcurrentHashMap、AtomicInteger、读写锁、写时复制思想


# 3.重构版
### —— 基于 UID 数据路由 + ThreadLocal 的无锁聚合方案。

##### 具体实现:
分层设计：query -> merge -> write。
- 实现逻辑解耦。
- 同时每层的需求下线程池设计不同，做到了不同任务场景对应不同的线程池管理。

| 层级     | query层                     | merger层                                          | write层                                            |
|--------|----------------------------|--------------------------------------------------|---------------------------------------------------|
| 职责     | 负责数据填充、校验等业务逻辑<br>需要多次调用外部服务 | 负责在内存中聚合订单数<br>定时任务触发落库逻辑                        | 负责落库具体逻辑                                          |
| 技术点    | CompletableFuture          | 无锁化 (Lock-Free)<br>ThreadLocal聚合<br>线程绑定<br>数据路由 | 批量写入<br>insert into on duplicate key update语法<br> |
| 任务类型   | 网络I/O阻塞                    | CPU密集计算                                          | 磁盘I/O阻塞                                           |
| 线程数    | 24<br>2倍cpu核数              | 长度12的单线程池数组<br>（与cpu核数相同）<br>按UID定向分发到线程         | 1<br>考虑DB压力                                       |
| 阻塞队列长度 | 60                         | 20                                               | 12                                                |
| 拒绝策略   | CallerRunsPolicy<br>降低MQ拉取速度 | AbortPolicy<br>触发上游阻塞重试逻辑                        | AbortPolicy<br>上游继续聚合<br>略微增大数据延迟                 |
| 吞吐量    | 120个/s                     |                                                  | 4000条/s                                           |

##### 聚合逻辑(Merge层)对比上一版设计有了很大改进。
- 线程在ThreadLocal内直接使用HashMap聚合数据，属于无锁化编程，代替了上一版ConcurrentHashMap和AtomicInteger。
- 通过UID哈希路由，做到了同一用户任务由同一线程执行，属于数据局部性优化，提高了聚合率。
##### 实现线程池优雅关闭，保证服务关闭时数据正确性。
- 使用注解@PreDestroy自定义逻辑。
- 线程池拒绝接收任务后，会等待任务完成即完成内存中数据落库再销毁。

# 4. 方案对比

| 维度    | 方案一               | 方案二                  |
| ----- | ----------------- | -------------------- |
| 并发控制  | 读写锁               | ThreadLocal 无锁并发     |
| 数据结构  | ConcurrentHashMap | HashMap              |
| IO处理  | 阻塞调用              | CompletableFuture 编排 |
| 线程池模型 | 单一线程池             | 按任务场景拆分              |
